# ADR-001: Share project memory through a committed markdown-per-fact mirror

## Status

Accepted

## Date

2026-06-30

## Context

The per-project memory store (`internal/core/projectmemory`) persists curated facts in a
SQLite database at `.rc/memory.db`. That file is binary and is listed in `.gitignore`
(`.gitignore:29`), so the memory is local to one machine and never shared across a team or
across clones. The package documents the database as "the single source of truth; there is
no markdown mirror" (`internal/core/projectmemory/store.go:1`).

The product goal is now the opposite: project memory should be shared across machines and
teammates through the repository, while the tool stays single-binary and local-first
(per `CLAUDE.md` runtime discipline). A binary SQLite file cannot be merged by git, so a
shareable representation is required.

## Decision

Add an explicit, versionable **text mirror** of the project memory:

- A new directory `.rc/memory/` holds one markdown file per memory, with YAML frontmatter
  for structured fields (`id`, `scope`, `key`, `title`, `tags`, `source`, `created_at`,
  `updated_at`) and the body as the file content.
- `.rc/memory/` is committed to git; `.rc/memory.db` stays gitignored and becomes a local
  cache/index rebuilt from the mirror.
- Two new subcommands drive sync explicitly: `rc memory export` (DB → mirror files) and
  `rc memory import` (mirror files → DB).
- Frontmatter is parsed with `gopkg.in/yaml.v3`, matching the existing frontmatter
  convention in `internal/core/tasks/parser.go`.

This supersedes the "no markdown mirror" tenet: the database remains the runtime query
engine (FTS5/BM25), but the committed mirror is the durable, shareable source of truth.

## Alternatives Considered

### Alternative 1: Commit the binary `.rc/memory.db`

- **Description**: Remove `memory.db` from `.gitignore` and version the SQLite file directly.
- **Pros**: Almost zero code; nothing new to build.
- **Cons**: SQLite files are binary and unmergeable; every `add`/`update` rewrites the file,
  producing constant churn and irresolvable merge conflicts with two or more contributors.
- **Why rejected**: Defeats the sharing goal — concurrent edits cannot be merged.

### Alternative 2: Single JSONL file (`.rc/memory.jsonl`)

- **Description**: Export all records to one line-delimited JSON file, committed to git.
- **Pros**: Trivial, exact round-trip of every field; least code.
- **Cons**: All facts live in one file, so simultaneous edits concentrate merge conflicts in
  that single file; less readable in pull-request review than per-fact files.
- **Why rejected**: Markdown-per-fact gives per-fact merge granularity (a conflict touches
  only the one fact edited) and readable diffs, which matters more than round-trip brevity.

## Consequences

### Positive

- Project memory is shareable through normal git workflow.
- Per-fact files mean merge conflicts are isolated to the single fact two people edited.
- Diffs are human-readable in code review.
- Mirrors the user's existing global-memory pattern (markdown files + index).

### Negative

- More code than committing the DB or a JSONL file: a writer, a parser, and two subcommands.
- The "single source of truth" shifts from the DB to the committed mirror; the DB is now a
  derived cache, which contributors must understand.

### Risks

- **Drift between DB and mirror** if a user edits one and forgets to sync. Mitigation: sync
  is explicit and documented; most-recent-wins on import (see ADR-002) reconciles content.
- **Frontmatter hand-edits** could malform a file. Mitigation: import validates required
  fields and reports the offending file without aborting the whole import.

## Implementation Notes

- Add export/import to the existing `internal/core/projectmemory` package (new file) and two
  subcommands in `internal/cli/memory.go`; do not introduce a new package.
- Export reads all records via the existing `Store.List` with an unbounded limit.
- No `.gitignore` change is needed: `.rc/memory.db` stays ignored; `.rc/memory/` is tracked.
