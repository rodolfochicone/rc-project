# ADR-002: Mirror identity and most-recent-wins conflict resolution

## Status

Accepted

## Date

2026-06-30

## Context

With a committed text mirror (ADR-001), the same logical fact can be edited on two machines
and reconciled through git. The sync must answer three questions: what gives a fact a stable
identity across machines, what the mirror filename is, and which version wins when the
database and an imported file disagree.

A memory record (`internal/core/projectmemory/store.go:38`) has a random `id`
(`mem-<hex>`, from `store.NewID`), an optional stable `key` unique within a `scope`, and
`created_at`/`updated_at` timestamps. The store already upserts keyed records by
`(scope, key)`, but `Add` always stamps `updated_at = now()` and unconditionally overwrites —
which is wrong for importing records whose timestamps must be preserved and compared.

## Decision

- **Identity**: a fact is identified by `(scope, key)` when `key` is set, otherwise by its
  `id`. Once a keyless fact's `id` is committed to the mirror, that `id` is its durable
  shared identity.
- **Filename**: `<scope>__<sanitized-key>.md` for keyed facts (so the same logical fact maps
  to the same file on every machine and git merges it in place), `<id>.md` for keyless facts.
  Sanitization lowercases and replaces any non-alphanumeric run with `-`.
- **Conflict resolution**: import is **most-recent-wins** by `updated_at`. For each imported
  record, the store inserts it when absent, updates the existing row only when the imported
  `updated_at` is strictly newer, and skips it otherwise. Imported `id`, `created_at`, and
  `updated_at` are preserved (not re-stamped).
- **Deletion propagation is out of scope for v1**: `export` writes/overwrites files but never
  deletes them, and `import` never deletes database rows for files that are absent. To remove
  a shared fact, a user deletes the `.md` file (and runs `rc memory delete` locally). This
  keeps v1 free of tombstone bookkeeping.

## Alternatives Considered

### Alternative 1: Identify every record by random `id` only

- **Description**: Use `<id>.md` for all facts and import purely by `id`.
- **Pros**: One uniform rule; no filename sanitization.
- **Cons**: The same keyed fact created independently on two machines gets two different
  random ids and two files, defeating in-place git merge for that fact.
- **Why rejected**: `(scope, key)` is the stable cross-machine identity the store already
  enforces; ignoring it produces duplicate files.

### Alternative 2: File-always-wins on import

- **Description**: Import overwrites the database unconditionally from the mirror.
- **Pros**: Simplest rule; mirror is strictly authoritative.
- **Cons**: A local edit not yet exported would be silently discarded on the next import.
- **Why rejected**: The user chose most-recent-wins to protect un-exported local edits.

## Consequences

### Positive

- Keyed facts map to a single stable file across machines, so git merges them in place.
- Most-recent-wins protects both local and remote edits without manual conflict steps.
- No tombstone machinery in v1.

### Negative

- Keyless facts depend on a random `id` for identity; two machines independently capturing the
  "same" keyless fact still produce two records (inherent — they were never linked).
- Deletions do not propagate automatically; a stale file re-imports a deleted fact until the
  file is removed.

### Risks

- **`updated_at` ties** (equal timestamps, divergent content): the existing row is kept
  (update only on strictly-newer). Acceptable; documented in the behavioral contract.
- **Clock skew** across machines could let an older edit win. Mitigation: timestamps are UTC
  (`store.FormatTimestamp`); skew is the same risk git itself carries and is accepted for v1.

## Implementation Notes

- Add a store method that upserts a fully-specified `Memory` (preserving id/timestamps) with a
  strictly-newer guard, distinct from `Add` (which stamps `now()`); run a batch import inside
  one transaction and return counts (added/updated/skipped).
- Filename sanitization must be deterministic so re-export is idempotent.
