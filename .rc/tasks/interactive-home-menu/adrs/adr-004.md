# ADR-004: Non-interactive realization — banner plus static enumerated menu

## Status

Accepted

## Date

2026-06-20

## Context

ADR-002 set the home menu as the universal default for bare `rc` but deferred the exact behavior when no interactive input device exists (piped output, CI, agent invocation). The current handler in `internal/cli/root.go` already distinguishes the two cases via `terminalWidth(cmd.OutOrStdout())`: a positive width means an interactive terminal, zero means non-interactive. The interactive menu cannot capture keystrokes without a TTY, and a menu that waits for input there would hang the caller.

## Decision

When `terminalWidth(...)` reports non-interactive (width `0`), render the banner followed by the menu options as a **static enumerated list** (e.g., `1. List Skills`, `2. Install RTK`, `3. Setup`, `4. Help`, `5. Quit`) and return immediately. No prompt, no blocking, no help fallback. When the terminal is interactive (width `> 0`), render the banner and enter the interactive huh-driven menu loop (ADR-003).

This honors the "always present the home menu" intent from ADR-002 while guaranteeing non-interactive callers never hang.

## Alternatives Considered

### Alternative 1: Static enumerated list with command hints

- **Description**: Same static list, but each line also prints the equivalent subcommand (e.g., `Setup — rc setup`).
- **Pros**: Slightly more useful for scripting.
- **Cons**: Adds noise; some items (List Skills, Quit) have no clean single-command equivalent.
- **Why rejected**: Marginal value; the plain enumerated menu satisfies the requirement. Can be revisited later.

### Alternative 2: Keep today's `cmd.Help()` fallback

- **Description**: Print help on non-TTY, as today.
- **Pros**: Matches clig.dev convention.
- **Cons**: Contradicts the chosen direction that the bare command always presents the home menu.
- **Why rejected**: Conflicts with ADR-002.

## Consequences

### Positive

- Non-interactive callers (CI, pipes, agents) see the menu contents and the command exits cleanly — zero hang risk.
- Consistent "menu is the face of bare `rc`" experience across contexts.

### Negative

- Non-interactive output is informational only; it cannot perform an action.

### Risks

- Reuse of the existing `terminalWidth` TTY signal must correctly classify edge cases (e.g., output to a non-`Fd` writer returns `0` → treated as non-interactive, which is the safe default).

## Implementation Notes

Branch on `terminalWidth(cmd.OutOrStdout())` in the bare-command `RunE`, replacing the current `cmd.Help()` fallback with the static enumerated menu renderer.
