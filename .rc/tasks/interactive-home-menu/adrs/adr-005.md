# ADR-005: Dispatch menu actions by re-invoking existing cobra subcommands

## Status

Accepted

## Date

2026-06-20

## Context

The home menu must perform five actions in-session: List Skills (read-only), Install RTK, Setup, Help, Quit. The codebase already implements most of this logic, but not as standalone reusable functions:

- `setup` is a cobra command whose logic lives in `setupCommandState.run(cmd, args)` (`internal/cli/setup.go`), tightly coupled to the cobra command and its flags.
- Help is available via `cmd.Help()`.
- There is **no** standalone command to "list skills" — the catalog loads via `loadEffectiveSetupCatalog()` / `setup.LoadCatalog` and `Skill{Name, Description}` carries the data.
- There is **no** standalone "install RTK" command — the primitives `setup.DetectRTK`, `setup.ResolveRTKInstall`, and `setup.RunRTKInstall` exist in `internal/setup/rtk.go` and are used inside the setup flow.

We must choose how the menu triggers these actions.

## Decision

Dispatch by **re-invoking the existing cobra subcommands** and adding small, local helpers only where no command exists:

- **Setup** and **Help**: locate the corresponding cobra command (e.g., `cmd.Root().Commands()` by name, or `cmd.Root().Help()`) and invoke it directly. No refactor of setup.
- **List Skills**: add a small read-only helper in `internal/cli` that calls the existing catalog loader and prints `Name` + `Description`. No installation.
- **Install RTK**: add a small helper in `internal/cli` that composes the existing `internal/setup/rtk.go` primitives (detect → resolve → confirm via huh → `RunRTKInstall`).

Do not extract setup into new reusable functions.

## Alternatives Considered

### Alternative 1: Extract reusable functions from setup/RTK

- **Description**: Refactor setup and RTK logic into standalone functions called by both the cobra commands and the menu.
- **Pros**: Cleaner long-term seams; avoids command-object plumbing.
- **Cons**: Modifies code that currently works; larger blast radius and risk; out of scope for a five-item menu.
- **Why rejected**: Violates surgical-change and YAGNI principles; the re-dispatch path reuses tested behavior with minimal new code.

## Consequences

### Positive

- Minimal, surgical change; reuses tested setup/help behavior as-is.
- New code limited to two small helpers (list skills, install RTK).

### Negative

- The menu depends on the cobra command graph being reachable from the bare-command handler (acceptable — it is the same root command).

### Risks

- Re-invoking a subcommand's `RunE` requires passing a valid `*cobra.Command`; the menu must pass the correct command/context so output and signal handling behave as in direct invocation. Mitigation: invoke through the resolved subcommand object so it carries its own flags and IO streams.

## Implementation Notes

The menu dispatcher maps each selected option to one of: a resolved cobra subcommand invocation (Setup, Help), a local helper (List Skills, Install RTK), or loop exit (Quit). Lives alongside the menu loop in `internal/cli`.
