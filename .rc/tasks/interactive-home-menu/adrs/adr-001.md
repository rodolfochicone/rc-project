# ADR-001: Integrated home menu as the default entry point for bare `rc`

## Status

Accepted

## Date

2026-06-20

## Context

Running `rc` with no subcommand currently renders a welcome banner (wordmark + tips + status footer) on an interactive terminal, and falls back to help/usage when output is not a TTY. New and recurring users have no guided, browsable way to reach the most common actions (listing skills, installing RTK, running setup, getting help) without already knowing the subcommand names.

We need to decide the product shape of a new interactive home menu attached to the bare command, balancing discoverability for newcomers against friction for power users.

## Decision

Adopt an **integrated home menu**: the bare `rc` splash keeps the existing banner on top and adds a navigable selection list directly below it, in the same screen. The list contains exactly five items for the MVP — List Skills, Install RTK, Setup, Help, Quit — and each selection runs its action within the same session. Navigation supports arrow keys and j/k, Enter to confirm, Esc/q to quit, and number keys as quick-select. "List Skills" is read-only. Named subcommands (`rc setup`, `rc add skill`, etc.) continue to work directly and bypass the menu, so the menu is a purely additive entry point on the zero-argument path.

## Alternatives Considered

### Alternative 1: Minimal launcher

- **Description**: Same menu but without number quick-select or richer affordances — only arrows/Enter/Esc.
- **Pros**: Faster to deliver; smaller surface area.
- **Cons**: Weaker discoverability and flow; defers UX wins that are cheap to add now.
- **Why rejected**: The incremental cost of quick-select and clear affordances is low and directly serves the dual onboarding + daily-use goal.

### Alternative 2: Extensible hub

- **Description**: Design the menu as a registry of items mapping to commands, pre-built to grow with future actions (tasks, reviews, ext).
- **Pros**: Easy to extend later; uniform item model.
- **Cons**: More scope and design decisions now for a five-item menu; risks over-engineering.
- **Why rejected**: YAGNI — the MVP needs five fixed items; a registry abstraction is not justified for current scope.

## Consequences

### Positive

- New users discover core capabilities without memorizing subcommands.
- Recurring users get a quick hub on the bare command.
- Preserves the established rc brand banner and visual identity.

### Negative

- The bare command becomes interactive by default, adding one step for experts who previously saw only the banner (mitigated: direct subcommands still bypass the menu).

### Risks

- Keyboard navigation requires an interactive terminal; behavior in non-interactive contexts is deferred to [ADR-002](adr-002.md) and the Risks/Open Questions sections of the PRD.

## Implementation Notes

Deferred to the TechSpec. Existing Bubble Tea and Huh infrastructure and the rc brand theme are available in the codebase.
