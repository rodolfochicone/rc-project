# ADR-001: Distribute rc skills and commands as a Claude Code plugin hosted in rc-project

- **Status:** Accepted
- **Date:** 2026-06-25

## Context

The bundled `rc-*` skills (`skills/`) and slash commands (`commands/`) are embedded
into the `rc` binary via `skills/embed.go` and `commands/embed.go`, and installed into
agent directories by `rc setup` (copy or symlink) with `rc sync` to refresh them.
"Updating" the skills therefore requires updating the `rc` binary. We want a
Claude-Code-native, auto-updatable distribution channel that is decoupled from the binary.

Claude Code plugins natively package both `skills/` (directories with `SKILL.md`) and
`commands/` (flat `.md` files) and are distributed through a `marketplace.json` catalog
hosted in git, with built-in update via `/plugin marketplace update`. The repository's
existing root layout (`skills/` + `commands/`) already matches the plugin layout exactly.

## Decision

Turn the existing `rc-project` repository into a single-plugin marketplace:

- Add `.claude-plugin/plugin.json` at the repo root (plugin root = repo root), pointing
  at the already-present `skills/` and `commands/` directories. No content is duplicated.
- Add `.claude-plugin/marketplace.json` listing one plugin, `rc`, with `source: "."`.
- The `//go:embed` directives are unaffected — `.claude-plugin/` is outside their globs,
  and `skills/`/`commands/` keep serving both the binary embed and the plugin.

Users install with `/plugin marketplace add rodolfochicone/rc-project` then
`/plugin install rc@rc-project`.

## Alternatives Considered

- **Dedicated public distribution repo + CI mirror** (`rodolfochicone/rc-plugins`): a thin repo
  holding only manifests + mirrored `skills/`/`commands/`, synced on each rc release.
  Light (~300KB clone), no auth friction. Rejected because it introduces a sync pipeline
  and a second source of truth, violating the single-source-of-truth and simplicity goals.
- **Keep only `rc setup`** (status quo): rejected — updates stay coupled to the binary,
  which is the problem being solved.

## Consequences

- **Positive:** Single physical source of truth; zero sync automation; the repo layout
  already conforms, so the change is two manifest files plus docs.
- **Negative / accepted trade-off:** Installing the plugin clones the whole `rc-project`
  repository (~140MB of git history) to deliver ~300KB of skills/commands, and copies it
  into the user's `~/.claude/plugins/cache`, exposing the full Go source there.
- **Negative / accepted trade-off:** `rc-project` is private, so `/plugin marketplace add`
  and background auto-update require git auth (`GH_TOKEN` / configured credentials).
  Documented as a prerequisite.
- Plugin distribution serves Claude Code only; other agents keep using `rc setup`
  (see ADR-003 derived scope).
