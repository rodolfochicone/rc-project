# ADR-002: Pin plugin updates to release tags via the `version` field

- **Status:** Accepted
- **Date:** 2026-06-25

## Context

A Claude Code marketplace controls when users receive plugin updates through the
`version` field of each plugin entry. Two behaviors exist:

- With `version` set, users only receive an update when that string changes.
- With `version` omitted on a git-hosted marketplace, every commit counts as a new version.

`rc-project` already versions releases as `vX.Y.Z` git tags driven by the OSS GoReleaser
flow (`.goreleaser.oss.yml`), which is the only release path (`internal/version` defaults
to `dev` and is stamped at build time).

## Decision

Set an explicit `version` in the `rc` plugin entry, aligned to the release tag (e.g.
`0.14.0`). Users receive plugin updates only on tagged releases, after running
`/plugin marketplace update`. The version bump is performed as part of the OSS GoReleaser
release flow (`.goreleaser.oss.yml`), either manually before tagging or wired into that
flow.

## Alternatives Considered

- **Omit `version` (track every commit):** rejected — every push to the target branch
  would ship to users, including work-in-progress and unreleased skill edits, making the
  channel unstable.
- **Pin to a commit SHA per release:** rejected — equivalent stability to tag pinning but
  less readable and not aligned with the existing `vX.Y.Z` tag convention.

## Consequences

- **Positive:** Predictable, stable updates aligned with existing release cadence.
- **Negative / accepted trade-off:** The `version` in the manifest must be bumped on every
  release; if forgotten, users silently stop receiving updates. Mitigated by folding the
  bump into the release pipeline.
