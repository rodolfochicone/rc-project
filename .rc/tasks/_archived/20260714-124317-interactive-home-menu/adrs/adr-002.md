# ADR-002: Home menu as the universal default, even outside an interactive terminal

## Status

Accepted

## Date

2026-06-20

## Context

Industry convention (clig.dev) for subcommand-based CLIs is to print help on bare invocation and to detect a non-TTY (piped output, CI, agent-driven runs) to avoid launching a blocking interactive prompt that can hang forever. The product owner explicitly chose, instead, that the bare `rc` should present the navigable home menu as its default behavior rather than falling back to help.

This creates a tension: arrow-key navigation requires an interactive terminal with a real input device. In a genuine non-interactive context (e.g., `rc | cat`, CI without a TTY, agent invocation with no stdin), there is no way to capture keystrokes, and a menu that waits for input risks hanging the caller.

## Decision

Treat the home menu as the **universal default entry point** for the bare `rc` command, per the product owner's direction. The menu is what bare `rc` is intended to surface in all default cases. The precise, safe behavior when no interactive input device is available is recorded as a deferred design question (see Open Questions in the PRD): the product intent is "always present the home menu," and the TechSpec must define how that intent is honored without hanging non-interactive callers (for example, by still emitting the menu's contents while not blocking on input when no input device exists).

## Alternatives Considered

### Alternative 1: Fall back to existing help on non-TTY

- **Description**: Keep today's behavior — show help/usage when output is not a terminal.
- **Pros**: Matches clig.dev convention; guaranteed safe for CI and agents; zero hang risk.
- **Cons**: The bare command no longer consistently presents the home menu, contradicting the chosen product direction.
- **Why rejected**: The product owner explicitly wants the home menu to be the default the user sees, not a fallback to help.

### Alternative 2: Print the menu as static enumerated text on non-TTY

- **Description**: When not interactive, print the options as a numbered static list and exit.
- **Pros**: More informative than raw help; never hangs.
- **Cons**: Largely duplicates help; not the navigable experience requested.
- **Why rejected**: Considered and not selected as the primary direction; remains a candidate safe behavior to evaluate in the TechSpec.

## Consequences

### Positive

- The home menu is the consistent, intended face of the bare command, reinforcing discoverability and the dual onboarding + daily-use goal.

### Negative

- Requires careful handling so non-interactive callers (CI, pipes, agents) are never left hanging on input.

### Risks

- **Hang risk in non-interactive contexts**: a menu awaiting keystrokes with no input device can block CI/agents indefinitely. Mitigation: the TechSpec must define a non-blocking behavior for the no-input-device case while still honoring the "present the home menu" intent. Tracked in PRD Risks and Open Questions.

## Implementation Notes

Detection of input-device availability and the exact non-interactive behavior are TechSpec concerns and intentionally not prescribed here.
