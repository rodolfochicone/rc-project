# ADR-004: Redesigned Content Model with Typed ACP Content Blocks

## Status

Accepted

## Date

2026-04-01

## Context

The current output model is Claude-specific. `ClaudeMessage` in `run/logging.go` parses stream-json with hardcoded field paths (`msg.Message.Content[].Type`, `msg.Message.Usage`). `TokenUsage` tracks Claude-specific cache fields (`CacheCreationTokens`, `CacheReadTokens`, `Ephemeral5mTokens`, `Ephemeral1hTokens`). Agents that don't emit Claude-format JSON get raw text passthrough with no structured data.

ACP defines a richer content model with typed content blocks: text, tool_use, tool_result, diff, terminal_output, image. Session updates include structured usage data and session status. This content typing is one of ACP's key value propositions.

The question is whether to map ACP's model back to the existing Claude-specific types, or redesign the model to be ACP-native.

## Decision

Redesign the content model to be ACP-native. Define `ContentBlock` with a `ContentBlockType` discriminator and type-specific payloads. Replace `ClaudeMessage` and `TokenUsage` with ACP-aligned `SessionUpdate` and `Usage` types. Expose all content block types to the UI layer.

## Alternatives Considered

### Alternative 1: Map ACP to Existing Claude Types

- **Description**: Parse ACP session updates and convert them back to `ClaudeMessage` format so downstream code (logging, UI) doesn't change.
- **Pros**: Minimal changes to logging and UI code, lower risk.
- **Cons**: Lossy — ACP's typed content blocks (diffs, tool calls, terminal output) would be flattened to text. Defeats the purpose of adopting ACP. New ACP content types would require adapter changes.
- **Why rejected**: The whole point of ACP is structured, typed content. Mapping it back to an untyped model throws away the value.

### Alternative 2: Normalize to Text + Metadata

- **Description**: Convert all ACP content blocks to text strings with metadata tags, keeping the UI model simple.
- **Pros**: Simple UI, backward-compatible log format.
- **Cons**: Can't render diffs with syntax highlighting, can't distinguish tool calls from output, can't show images. Limits future UI improvements.
- **Why rejected**: User explicitly requested exposing all content block types to the UI for rich rendering.

## Consequences

### Positive

- All ACP content block types available for rich TUI rendering (syntax-highlighted diffs, tool call visualization, terminal output)
- Usage model matches ACP spec exactly — no lossy translation
- Model is agent-agnostic — any ACP-compatible agent's output renders correctly
- Future ACP content types can be added by extending the enum, not by rewriting parsing

### Negative

- UI layer (`ui_view.go`, `ui_model.go`) must handle multiple content block types instead of plain text lines
- More complex rendering logic in the Bubble Tea TUI
- Log files will contain structured content instead of plain text

### Risks

- UI complexity increase. Mitigation: implement a fallback text renderer for unknown block types; new types degrade gracefully to raw text.

## Implementation Notes

- New types in `internal/core/model/content.go`:
  - `ContentBlockType` enum (text, tool_use, tool_result, diff, terminal_output, image)
  - `ContentBlock` struct with `Type` + `Data json.RawMessage`
  - `SessionUpdate` struct with `Blocks []ContentBlock`, `Usage Usage`, `Status SessionStatus`
  - `Usage` struct aligned to ACP usage fields
- `ClaudeMessage` and current `TokenUsage` in `run/logging.go` and `run/types.go` will be removed
- UI messages change from `jobLogUpdateMsg{Out: []string, Err: []string}` to `jobUpdateMsg{Blocks: []ContentBlock}`
