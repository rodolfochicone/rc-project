# ADR-003: Subprocess-over-Stdio for ACP Transport

## Status

Accepted

## Date

2026-04-01

## Context

ACP supports two deployment models for agent communication:

1. **Subprocess**: rc spawns the agent (or ACP adapter) binary as a child process and speaks JSON-RPC 2.0 over stdin/stdout.
2. **Server**: An ACP-compatible agent runs as a long-lived server process, and rc connects via TCP/Unix socket.

The current architecture spawns agents as subprocesses via `exec.CommandContext()`. The project's runtime discipline states: "Keep the system single-binary and local-first. Introduce sidecars or external control planes only with a written techspec."

## Decision

Use the subprocess-over-stdio model. rc spawns the agent binary (or its ACP adapter) as a child process and communicates via JSON-RPC 2.0 over the process's stdin/stdout pipes.

## Alternatives Considered

### Alternative 1: Connect to Running ACP Server

- **Description**: Agents run as long-lived ACP servers. rc connects as a client via TCP or Unix socket.
- **Pros**: Agent can be shared across sessions, no process spawn overhead per job, supports remote agents.
- **Cons**: Requires external process management (systemd, Docker, manual start), breaks local-first philosophy, adds operational complexity, requires network configuration.
- **Why rejected**: Violates the project's runtime discipline of single-binary, local-first operation. Adds deployment complexity without clear benefit for rc's use case.

### Alternative 2: Embedded Agent (In-Process)

- **Description**: Import agent logic as a Go library and run in-process.
- **Pros**: No IPC overhead, simplest lifecycle management.
- **Cons**: Agents are written in different languages (Rust, Python, TypeScript), not embeddable as Go libraries. Fundamentally incompatible.
- **Why rejected**: Technically infeasible for the current agent ecosystem.

## Consequences

### Positive

- Matches current process model — minimal architectural change to execution pipeline
- Local-first, no external dependencies or servers to manage
- Process isolation: agent crashes don't take down rc
- Works with any ACP-compatible binary regardless of implementation language

### Negative

- Process spawn overhead per job (mitigated by ACP session reuse within a job)
- Cannot share agent state across jobs (each job gets a fresh process)

### Risks

- Large binary startup time for some agents. Mitigation: ACP session creation happens once per job; the protocol amortizes startup over the session.

## Implementation Notes

- `agent.Client` spawns the binary via `exec.CommandContext()`, wires `cmd.Stdin` and `cmd.Stdout` to JSON-RPC 2.0 codec
- `cmd.Stderr` is still captured for error logging (ACP protocol only uses stdin/stdout)
- Process lifecycle tied to `context.Context` cancellation — same pattern as today
- Working directory set to the repository root, same as current behavior
