# ADR-005: ACP Session Lifecycle for Timeout and Retry Management

## Status

Accepted

## Date

2026-04-01

## Context

The current timeout mechanism uses an `activityMonitor` in `logging.go` that records timestamps on every stdout write. A watchdog goroutine in `execution.go` polls `timeSinceLastActivity()` every 5 seconds and sends SIGTERM → SIGKILL if the timeout is exceeded. This is a heuristic: it measures "is stdout producing bytes" rather than "is the agent making progress."

ACP provides structured session lifecycle: session creation, streaming update notifications with status fields, heartbeats, and session completion events. The protocol knows when an agent is actively working, waiting for tools, or idle.

## Decision

Replace the `activityMonitor` with ACP session lifecycle events for timeout detection. Use ACP's session status and update notifications as the authoritative signal for agent liveness. Retry logic will use ACP session error/completion status rather than process exit codes.

## Alternatives Considered

### Alternative 1: Keep Activity Monitor, Feed from ACP Updates

- **Description**: Maintain the current `activityMonitor` pattern but call `recordActivity()` from ACP session update events instead of raw stdout bytes.
- **Pros**: Minimal change to timeout logic, proven pattern.
- **Cons**: Still a heuristic — ACP provides richer session state (working, idle, waiting_for_tool) that a simple "last activity" timestamp can't express. Misses protocol-native timeout/error semantics.
- **Why rejected**: ACP's session lifecycle provides strictly more information than activity timestamps. Using ACP status is both simpler and more accurate.

### Alternative 2: Dual Timeout (ACP + Activity Monitor)

- **Description**: Use ACP session status as primary, fall back to activity monitoring if ACP heartbeats stop.
- **Pros**: Defense in depth against ACP adapter bugs that stop sending heartbeats.
- **Cons**: Two timeout systems to reason about, potential for conflicting signals.
- **Why rejected**: Adds complexity. If the ACP adapter is broken enough to not send heartbeats, the session will eventually error out via process-level signals (context cancellation kills the subprocess).

## Consequences

### Positive

- Timeout based on semantic agent state, not byte-level output heuristics
- Retry decisions informed by structured error types (ACP error codes) rather than process exit codes
- Eliminates the 5-second polling loop — event-driven instead
- Session completion events provide clean shutdown without SIGTERM/SIGKILL sequences

### Negative

- Depends on ACP adapters correctly implementing session lifecycle events
- Process-level kill (SIGTERM/SIGKILL) still needed as a last resort if ACP session hangs

### Risks

- ACP adapter doesn't send heartbeats or status updates. Mitigation: keep a process-level context timeout as an ultimate backstop (context cancellation kills the subprocess). This is not the same as the activity monitor — it's a hard wall clock timeout, not a heuristic.

## Implementation Notes

- Remove `activityMonitor` struct and `newActivityMonitor()` from `logging.go`
- ACP `Session.Updates()` channel delivers `SessionUpdate` with status field
- Timeout logic in `execution.go` listens on `Session.Done()` channel + `ctx.Done()` for cancellation
- Process-level `context.WithTimeout()` serves as ultimate backstop
- Retry backoff logic reads `Session.Err()` for structured ACP error codes instead of parsing `exec.ExitError`
