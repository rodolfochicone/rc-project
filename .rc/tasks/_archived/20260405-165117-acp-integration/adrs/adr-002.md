# ADR-002: Vendor coder/acp-go-sdk as ACP Client Foundation

## Status

Accepted

## Date

2026-04-01

## Context

RC needs a JSON-RPC 2.0 client that speaks ACP protocol over stdio to communicate with coding agents. The options are: use an existing SDK, or implement the protocol from scratch.

`coder/acp-go-sdk` (128 stars, Apache 2.0, maintained by Coder) provides a Go implementation of the ACP client with JSON-RPC 2.0 transport, session management, and typed content block definitions. It is the only Go SDK in the ACP ecosystem.

## Decision

Vendor `coder/acp-go-sdk` via `go get` as a direct dependency. Wrap it behind RC's own `Client` and `Session` interfaces so the SDK can be swapped if needed without changing consumers.

## Alternatives Considered

### Alternative 1: Implement ACP Client from Scratch

- **Description**: Build a custom JSON-RPC 2.0 client that implements only the ACP subset RC needs.
- **Pros**: No external dependency, full control over implementation, minimal footprint.
- **Cons**: Significant development effort for JSON-RPC 2.0 framing, error handling, and protocol edge cases. Risk of subtle protocol deviations. Must track ACP spec changes manually.
- **Why rejected**: The protocol has enough complexity (session lifecycle, content block typing, notification streaming) that a proven implementation saves weeks of work and reduces bug surface.

### Alternative 2: Use a Generic JSON-RPC 2.0 Library + Custom ACP Layer

- **Description**: Use a library like `sourcegraph/jsonrpc2` for transport and build ACP session semantics on top.
- **Pros**: Decoupled from Coder's SDK lifecycle, flexible transport layer.
- **Cons**: Still need to implement all ACP-specific types, session management, and content block parsing. Most of the complexity is ACP-specific, not JSON-RPC-generic.
- **Why rejected**: The ACP-specific logic is the bulk of the work; a generic JSON-RPC library only handles the easy part.

## Consequences

### Positive

- Proven JSON-RPC 2.0 + ACP implementation, already tested against 15+ agents via ACPX
- Apache 2.0 license is compatible with RC's licensing
- Type definitions for all ACP content blocks included
- Wrapping behind our own interface isolates consumers from SDK internals

### Negative

- Dependency on Coder's maintenance cadence — if SDK is abandoned, we own the fork
- SDK may include features RC doesn't need, adding binary size

### Risks

- SDK is relatively young (128 stars). Mitigation: pin to a specific version, wrap behind interfaces, keep the option to fork or replace open.
- Breaking changes in SDK API. Mitigation: the wrapper interface absorbs API changes without propagating to RC internals.

## Implementation Notes

- Install via `go get github.com/coder/acp-go-sdk@<pinned-version>`
- Define `agent.Client` and `agent.Session` interfaces that wrap SDK types
- SDK types used internally within `internal/core/agent/` only — never leaked to `model/` or `run/` packages
