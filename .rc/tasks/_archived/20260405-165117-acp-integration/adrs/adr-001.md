# ADR-001: Full Replacement of Legacy Agent Drivers with ACP

## Status

Accepted

## Date

2026-04-01

## Context

rc currently communicates with 6 coding agents (Claude, Codex, Droid, Cursor, OpenCode, Pi) via a custom `spec` map registry in `internal/core/agent/ide.go`. Each agent has handcrafted `commandFunc` and `shellPreviewFunc` implementations that build `*exec.Cmd` with agent-specific CLI flags. Output parsing in `logging.go` branches on IDE type to decide between JSON formatting (Claude/Droid/Cursor/OpenCode/Pi) and line filtering (Codex).

Adding a new agent requires touching 4+ files: constants in `model.go`, spec entry in `ide.go`, command builder functions, and conditional output handling in `logging.go`. The output format is Claude-specific (`stream-json` with `ClaudeMessage` structs), and agents that don't match this format get raw passthrough.

ACP (Agent Client Protocol) is a JSON-RPC 2.0 protocol over stdio that standardizes communication with coding agents. It has 34+ registered agents with adapters and a Go SDK (`coder/acp-go-sdk`).

The question is whether ACP should coexist as an alternative transport alongside the legacy approach, or fully replace it.

## Decision

ACP will fully replace the current CLI subprocess driver approach. There will be no legacy fallback or `--ide-transport` flag. All 6 agents will communicate exclusively via ACP.

## Alternatives Considered

### Alternative 1: Dual Transport (ACP + Legacy)

- **Description**: Keep the current `spec` map for legacy invocation and add ACP as an opt-in transport via a flag like `--transport acp`.
- **Pros**: Lower risk, incremental migration, agents with immature ACP adapters can fall back.
- **Cons**: Two code paths to maintain, doubled testing surface, feature parity drift between transports, conditional branching throughout execution pipeline.
- **Why rejected**: The maintenance burden of two transport layers outweighs the migration risk. ACP adapters exist for all 6 current agents.

### Alternative 2: ACP for New Agents Only

- **Description**: Keep legacy drivers for existing agents, use ACP only for newly added agents.
- **Pros**: Zero risk to existing functionality.
- **Cons**: Permanent architectural split, new agents get richer features (typed content blocks) that legacy agents can never use, increasing divergence over time.
- **Why rejected**: Creates a permanent two-class system that gets worse with every new agent.

## Consequences

### Positive

- Single code path for all agent communication
- Adding new agents requires only an `AgentSpec` entry (binary, bootstrap args, env vars)
- Typed content blocks (diffs, tool calls, terminal output) available for all agents uniformly
- Protocol-native session lifecycle, timeout, and usage tracking

### Negative

- All 6 agents must work via ACP before the feature can ship — no partial rollout
- Breaking change for any users who depend on the current CLI flag signatures or output format

### Risks

- Some ACP adapters (Cursor, Pi) may be immature. Mitigation: verify each adapter in isolation before integration, document minimum adapter versions.
- All-or-nothing release increases blast radius. Mitigation: thorough mock-based testing plus real agent smoke tests before merge.

## Implementation Notes

- The `specs` map in `ide.go`, all `buildXxxCommand`/`xxxCommand` functions, and `ide_helpers.go` will be removed entirely.
- IDE constants in `model.go` (`IDECodex`, `IDEClaude`, etc.) will be preserved as registry keys.
- The `jsonFormatter` and `activityMonitor` in `logging.go` will be replaced by ACP session update processing.
