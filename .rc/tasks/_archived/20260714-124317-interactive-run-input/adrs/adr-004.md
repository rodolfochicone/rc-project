# ADR-004: Interactivity is opt-in via a flag set at run start

## Status

Accepted

## Date

2026-06-14

## Context

Today `RequestPermission` auto-approves and exec runs finalize at `end_turn`
([ADR-002](adr-002.md)). If every run blocked waiting for input, headless and
automated/scheduled runs would hang indefinitely, since no one is attached to
answer. The PRD scopes the MVP to user-observed runs and explicitly defers
headless handling to a later phase.

## Decision

Interactivity is **opt-in**. A boolean `interactive` flag is set on the run start
request (carried by the exec/run start path, originating from the escale UI). The
flag is stored on the run and threaded down to the executor and ACP client via
`model.RuntimeConfig`.

- When `interactive` is true: `RequestPermission` blocks for a user decision and
  the exec loop pauses on `end_turn` for a user answer.
- When `interactive` is false (default): existing behavior is unchanged —
  permissions auto-approve and runs finalize at `end_turn`.

## Alternatives Considered

### Alternative 1: All runs interactive

- **Description**: Always block for permissions and pause on `end_turn`.
- **Pros**: No flag to thread; one code path.
- **Cons**: Headless/scheduled/automated runs hang forever waiting for input that
  never comes; breaks current non-interactive behavior.
- **Why rejected**: Unacceptable regression for automated runs.

### Alternative 2: Detect "someone is attached" (live stream subscriber)

- **Description**: Block only when a UI client is subscribed to the run stream.
- **Pros**: No explicit flag; "magic" correctness.
- **Cons**: Requires reliable attach/detach detection; racy (subscriber drops
  mid-prompt); harder to reason about. PRD lists this as a Phase 2 option.
- **Why rejected**: Too implicit for the MVP; deferred to Phase 2.

## Consequences

### Positive

- Automated runs keep working untouched; only explicitly interactive runs block.
- Clear, testable switch.

### Negative

- Callers must set the flag; a run started without it cannot become interactive
  mid-flight.

### Risks

- A user starts an interactive run and walks away, holding a session open
  indefinitely. Accepted for MVP ("wait indefinitely"); cancel reclaims it.

## Implementation Notes

Add `interactive` to the exec/start request schema (OpenAPI) and
`model.RuntimeConfig`; gate the new behavior on it in `internal/core/agent`
(`RequestPermission`) and `internal/core/run/exec` (turn loop).
