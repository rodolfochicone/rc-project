# ADR-002: Block-in-callback permissions + live-session re-prompt for multi-turn

## Status

Accepted

## Date

2026-06-14

## Context

Two pause moments must be supported (see [ADR-001](adr-001.md)) and they are
mechanically different in the ACP client (`internal/core/agent/client.go`):

- **Permission requests**: `RequestPermission` is invoked **synchronously by the
  ACP SDK while `c.conn.Prompt()` is still in flight** (client.go:523). Today it
  auto-approves the first option. If the callback blocks, the agent's turn blocks
  with it.
- **Skill questions**: the agent writes the question as a message and the turn
  ends — `c.conn.Prompt()` returns `StopReason=end_turn` and `runPrompt` calls
  `session.finish(StatusCompleted)` (client.go:785-822). Continuing requires a new
  prompt on the session.

A key fact from exploration: the ACP connection (`c.conn`) and the session object
stay alive until the job closes the client (`defer state.close()` in
`exec.go:235`). So a follow-up prompt can be sent on the **same live session**
without persisting and reloading it.

## Decision

Use two complementary mechanisms, both gated by the interactive flag
([ADR-004](adr-004.md)):

1. **Permissions — block in the callback.** When interactive, `RequestPermission`
   emits an `awaiting_input` event and blocks on the run's input coordinator until
   the user picks an option (or cancels), then returns that outcome. The agent's
   current turn continues — no resume needed.
2. **Skill questions — re-prompt the live session.** Add a method to continue the
   existing live session with a new prompt (e.g., `Session.Continue(ctx, prompt)`
   that calls `c.conn.Prompt()` again on the same `SessionId`). The exec executor,
   for an interactive run, does not finalize on `end_turn`: it emits
   `awaiting_input`, blocks on the coordinator, and on response re-prompts the
   live session with the user's answer. The loop ends when the user cancels.

## Alternatives Considered

### Alternative 1: Persisted ResumeSession per turn

- **Description**: After each `end_turn`, close the session and use
  `ResumeSession`/LoadSession to start the next turn from persisted state.
- **Pros**: Reuses existing resume path; survives process restarts.
- **Cons**: Depends on agent `LoadSessionSupported`; tears down and reloads the
  connection every turn; more moving parts and failure modes.
- **Why rejected**: The live connection is already open; re-prompting it is
  simpler and removes the LoadSession dependency for the MVP.

### Alternative 2: Everything through the permission handshake

- **Description**: Require the agent to ask questions via a tool/permission so all
  pauses are block-in-callback and `end_turn` always means done.
- **Pros**: One mechanism; no multi-turn loop.
- **Cons**: RC's PRD/TechSpec skills emit plain-text questions and stop — they do
  not use a permission handshake. Would require changing every skill.
- **Why rejected**: Does not match how skills actually pause today.

## Consequences

### Positive

- Permissions resolve within the same turn; no lifecycle change for that case.
- Multi-turn questions work in-process with no persisted-resume dependency.

### Negative

- The exec executor gains a turn loop and must hold the client/session open across
  turns instead of one-shot.
- For MVP, an interactive run does not reach `completed` naturally — every
  `end_turn` pauses; the run ends when the user cancels (acceptable trade-off;
  see Known Risks / Open Questions in the TechSpec).

### Risks

- Blocking the ACP callback indefinitely ties up the connection. Mitigation: the
  block is always cancellable via the run context (cancel ends the wait).
- Re-prompting the live session must stream into the same session updates channel.
  Mitigation: reuse the existing `SessionUpdate` routing keyed by `SessionId`.

## Implementation Notes

`RequestPermission` and a new `Session.Continue` (or `Client.Prompt` on a live
session id) in `internal/core/agent`; turn loop in `internal/core/run/exec`. The
input coordinator is threaded via `model.RuntimeConfig` / `SessionRequest`.
