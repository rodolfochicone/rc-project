# ADR-003: Represent "awaiting input" as an event plus a snapshot field

## Status

Accepted

## Date

2026-06-14

## Context

The run must surface a clear "waiting on you" state ([ADR-001](adr-001.md)). Run
status today is a small enum consumed widely: terminal-status checks
(`isTerminalRunStatus`), run-list status filters, the OpenAPI `Run` schema, the
global DB, and frontend status tones. "Awaiting input" is a transient, frequently
toggled state, not a terminal outcome.

## Decision

Do not add a new top-level run status. Instead:

- Emit a new event `session.awaiting_input` carrying the pending prompt (kind =
  `permission` | `question`, the prompt text, a correlation id, and — for
  permissions — the ACP-supplied options). Clear it with a follow-up signal when
  the user responds (the next session update, or a dedicated resolved marker).
- Expose a `pending_input` field on the run snapshot payload, set while a prompt
  is outstanding and cleared on response.

The run keeps status `running`. The UI derives the waiting affordance from the
live `awaiting_input` event and/or `snapshot.pending_input`, in both the run
detail and (optionally) the runs list.

## Alternatives Considered

### Alternative 1: New run status `waiting` / `awaiting_input`

- **Description**: Add a first-class status value.
- **Pros**: Directly filterable; explicit in every status-aware surface.
- **Cons**: Wide blast radius — status enum, terminal checks, list filters,
  OpenAPI `Run` schema, DB writes, frontend tones — for a transient state that
  toggles many times within one run.
- **Why rejected**: Cost and risk outweigh the benefit for the MVP; an event plus
  snapshot field gives the UI everything it needs.

## Consequences

### Positive

- No churn in status enums or terminal-status logic; the run stays `running`.
- The pending prompt and its options travel in one typed payload the UI renders
  directly.

### Negative

- "Waiting" is not a queryable status; finding waiting runs relies on the snapshot
  field rather than a status filter.

### Risks

- Stale `pending_input` if the clear signal is missed. Mitigation: clear on the
  next session update and on run termination; treat a newer event as superseding.

## Implementation Notes

`EventKindSessionAwaitingInput` in `pkg/rc/events`, payload in
`pkg/rc/events/kinds/session.go`; `pending_input` added to the run snapshot
payload and OpenAPI schema (codegen regenerates the TS types).
