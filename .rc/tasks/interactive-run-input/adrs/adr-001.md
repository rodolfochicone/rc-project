# ADR-001: Pause-and-resume interactive runs inside the run detail view

## Status

Accepted

## Date

2026-06-14

## Context

rc runs driven by AI agents over ACP regularly reach points where the agent
needs the user before it can continue. Two distinct moments cause this:

1. **Permission requests** — the agent asks to run a sensitive action (write a
   file, run a command). ACP defines a native `session/request_permission`
   handshake for this. Today rc auto-approves the first option, so the user
   never sees the choice.
2. **Skill questions** — workflow skills such as PRD and TechSpec creation are
   designed to ask the user a question (often with labeled options A/B/C/D) and
   then end the turn, expecting a human answer before proceeding.

The escale web UI currently observes runs read-only: it streams events over SSE
and the only write action is `POST /runs/:id/cancel`. There is no way for a user
watching a run to answer the agent. The practical consequence is that the most
valuable interactive workflows (ideation, spec clarification, action approval)
cannot be driven from the web UI at all.

We need a product model for letting the user respond to a running agent.

## Decision

Introduce a first-class **"Awaiting your response"** run state, surfaced and
answered inside the existing run detail view. When the agent pauses — whether for
a permission request or a skill question — the run detail shows the prompt with a
response affordance (clickable buttons when discrete options are available, plus
a free-text box), the user answers, and the same run **resumes**. Both
interaction types share one mental model and one surface. A paused run waits
indefinitely until the user responds or cancels.

## Alternatives Considered

### Alternative 1: Approval-gates first, questions later (phased)

- **Description**: Ship only the ACP permission-approval prompt in the MVP
  (replace auto-approve with a UI choice). Defer free-text skill-question
  answering to a later phase.
- **Pros**: Lower risk; uses the native ACP permission handshake; faster first
  value.
- **Cons**: Does not solve the PRD/TechSpec clarification pain that motivated the
  request; leaves the primary user need unmet at MVP.
- **Why rejected**: The originating need is answering skill questions; shipping
  approvals alone would feel like the wrong feature first.

### Alternative 2: Dedicated interactive session console

- **Description**: A new chat-style surface, separate from the observational run
  detail, where the user conducts a full back-and-forth with the agent.
- **Pros**: Most powerful; supports open-ended conversation.
- **Cons**: Much larger scope; overlaps and competes with the existing `exec`
  single-prompt system; duplicates the run detail experience and splits where
  users look for a run.
- **Why rejected**: Too large for the need and introduces a second, redundant
  place to watch and drive a run.

## Consequences

### Positive

- One consistent model for both permission approvals and skill questions.
- Reuses the run detail page and the existing live event stream the user already
  watches.
- Unlocks the full PRD/TechSpec/ideation workflows from the web UI.

### Negative

- Requires the run/session lifecycle to stay alive across turns instead of
  terminating after one prompt — the most sensitive change, deferred to the
  TechSpec.
- A forgotten paused run holds a live session open indefinitely (accepted
  trade-off for MVP simplicity).

### Risks

- **Detecting discrete options from free-text skill questions is heuristic and
  fragile.** Mitigation: always provide the free-text box as a fallback so the
  user can answer even when option buttons are not detected.
- **Headless/automated runs could hang waiting for input.** Mitigation: treated
  as an Open Question for the TechSpec; the MVP targets user-observed runs and
  keeps "wait indefinitely" semantics.

## Implementation Notes

Deferred to the TechSpec. Known integration points from codebase exploration:
the `RunManager.active[runID]` registry (mirrors how cancel reaches a live run),
a new run write endpoint analogous to `POST /runs/:id/cancel`, keeping the ACP
session reachable between turns, and the `RequestPermission` / `ResumeSession`
paths in the ACP client.
